# 13 GitHub 安全审计报告

文档版本：v0.1  
日期：2026-05-01

## 1. 审计范围

本轮审计覆盖 GitHub 展示安全相关的四类内容：

- 当前 `main` 最新版本中的代码、文档、数据文件。
- Git 历史提交中的路径、密钥、证书和敏感标记。
- `.gitignore` 是否能挡住本机私有配置、快照和 Xcode 用户状态。
- 仓库中是否误提交 PDF、Office 文档、压缩包、证书或异常大文件。

## 2. 当前版本结论

当前 `main` 最新版本未发现高危泄露。

已确认：

- 未发现 API Key、GitHub Token、私钥、云服务访问密钥等凭据样式字符串。
- 未发现本机真实绝对路径。
- 未跟踪 `.local` 私有同步配置或私有快照。
- 未跟踪 Xcode 用户状态文件。
- 未跟踪 PDF、Word、Excel、压缩包、证书或 provisioning profile。
- 最大文件为 App 图标 PNG，属于正常展示资产。

## 3. 已发现隐患

Git 历史中有两个旧提交曾出现本机绝对路径：

- `774d9ae`：真实考试规划同步阶段。
- `944126b`：App 图标与 A4 计划同步阶段。

涉及文件类型：

- 结构化种子数据。
- 同步说明文档。
- 项目进展日志。

风险判断：

- 风险级别：中等。
- 泄露类型：本机路径元数据。
- 未发现密钥、账号密码、证书、私钥或完整复习资料正文泄露。
- 如果仓库作为校招作品集公开展示，建议清理 Git 历史，避免暴露个人目录结构和资料组织方式。

## 4. 推荐修复方案

要彻底清理 GitHub 页面上的历史记录，需要执行历史改写：

1. 将旧提交中的本机绝对路径替换为通用描述。
2. 重写本地 Git 历史。
3. 再次扫描全历史，确认无路径和密钥命中。
4. 使用 `--force-with-lease` 推送到 GitHub。

注意：

- 这是 Git 历史改写，会改变旧 commit hash。
- 如果别人已经基于当前仓库拉取代码，需要重新同步。
- GitHub 可能短时间保留缓存或孤立对象，但正常分支历史会变干净。

## 5. 后续安全规则

后续同步必须遵守：

- 真实本机路径只放在 `.local.json`。
- 公开 JSON 只保留脱敏来源名称和统计结果。
- 不提交 PDF、原始课件、截图、Office 文档和压缩包，除非先做脱敏审查。
- 不提交 `.env`、证书、私钥、profile、token。
- 不提交 Xcode `xcuserdata` 和 `xcuserstate`。
- 每次推送前执行一次当前版本扫描和历史敏感词扫描。

## 6. 当前建议

当前最新版可以继续本地开发和演示，但在把 GitHub 仓库公开投递给校招面试官之前，建议先完成一次历史清洗和强制安全推送。
