## 1. 为什么用 Core Data 而不是 SwiftData？

### 1.1 架构差异对比

| 维度 | Core Data | SwiftData |
|------|-----------|-----------|
| **API 层** | Objective-C 根底，C 风格 API | 纯 Swift，@Model 宏驱动 |
| **模型定义** | `.xcdatamodeld` 可视化编辑器 | 代码内 `@Model` 装饰类 |
| **线程安全** | `NSManagedObjectContext` 绑定线程 | `@ModelActor` 自动隔离 |
| **关系映射** | `NSSet` + `@NSManaged` | 原生 Swift 类型自动推断 |
| **迁移策略** | 手动 Lightweight/Mapping Model | 自动轻量迁移 |
| **最低版本** | iOS 3.0+ | iOS 17.0+ |
| **与 UIKit 集成** | `NSFetchedResultsController` 成熟 | 与 SwiftUI 深度绑定 |

**[原理] 技术决策的本质**：SwiftData 是 Core Data 的 Swift 原生封装层，底层仍依赖 SQLite。它用宏（`@Model`、`@Relationship`）和属性包装器（`@Query`）替代了 `.xcdatamodeld` 和 `NSFetchRequest`，但**并非全新的存储引擎**。

```swift
// [原理] Core Data 显式声明模型关系，精确控制内存和线程
@objc(CourseEntity)
public class CourseEntity: NSManagedObject {
    @NSManaged public var id: String
    @NSManaged public var name: String
    @NSManaged public var knowledgePoints: NSSet?  // [原理] NSSet 是无序集合，Core Data 用它实现多对多关系
    @NSManaged public var questions: NSSet?
}

// [设计复盘] 对比 SwiftData：@Model class Course { @Relationship var knowledgePoints: [KnowledgePoint] }
// 设计复盘回答要点：SwiftData 的 @Relationship 底层仍转化为 Core Data 的 Relationship 描述，
// 只是编译器在宏展开阶段自动生成了 NSManagedObject 子类代码。
```

### 1.2 本项目选择 Core Data 的核心原因

**原因一：部署目标兼容性**

```swift
// [原理] FinalPilot 支持 iOS 16+，SwiftData 要求 iOS 17+。在 2024-2025 年里程碑场景中，
// 读者关注"如何支持旧版本用户"，Core Data 是更稳妥的项目说明故事。
// deployment target: iOS 16.0
```

**原因二：手动控制迁移策略**

```swift
// [原理] CoreDataMigrationHelper 实现了"从 SeedData 到持久化存储"的首次迁移，
// 这是 SwiftData 自动迁移无法覆盖的场景（硬编码数据初始化）。
final class CoreDataMigrationHelper {
    static func migrateSeedDataIfNeeded(context: NSManagedObjectContext) {
        let fetchRequest: NSFetchRequest<CourseEntity> = CourseEntity.fetchRequest()
        fetchRequest.fetchLimit = 1
        
        // [原理] count(for:) 比 fetch 后判空更高效：SQLite 层面执行 COUNT(*) 而非加载行数据
        let count = (try? context.count(for: fetchRequest)) ?? 0
        guard count == 0 else { return }  // [设计复盘] 为什么用 count 而不是 fetch + isEmpty？
        
        // 首次启动：将 SeedData 硬编码数据批量写入 Core Data
        for course in SeedData.courses {
            _ = CourseEntity.fromCourse(course, context: context)
        }
        // ... 其他实体迁移
        
        try? context.save()
    }
}
```

**[设计复盘] 高频问题**："如果未来要迁移到 SwiftData，怎么做？"

> 答：Core Data 和 SwiftData 可以**共存**。SwiftData 的 `ModelContext` 底层仍读写同一个 `.sqlite` 文件。迁移策略是：
> 1. 保持现有 Core Data 栈不变
> 2. 新增 SwiftData 模块，使用 `.storeIn(groupContainer:)` 指向同一数据库文件
> 3. 通过 `NSPersistentContainer` 的 `viewContext` 和 SwiftData 的 `modelContext` 做双写过渡
> 4. 验证数据一致性后，逐步淘汰 Core Data 代码

### 1.3 线程安全与上下文策略

```swift
final class DataController: ObservableObject {
    let container: NSPersistentContainer
    
    var viewContext: NSManagedObjectContext {
        container.viewContext
    }
    
    init(inMemory: Bool = false) {
        // [原理] mergeByPropertyObjectTrump：当内存对象和持久化对象冲突时，以内存对象为准。
        // 这是"离线优先"策略：用户在飞行模式下修改的数据，联网同步时不被服务器覆盖。
        container.viewContext.mergePolicy = NSMergePolicy.mergeByPropertyObjectTrump
        
        // [原理] automaticallyMergesChangesFromParent = true：
        // 后台上下文执行 save() 后，viewContext 自动合并变更，无需手动刷新。
        container.viewContext.automaticallyMergesChangesFromParent = true
    }
    
    func newBackgroundContext() -> NSManagedObjectContext {
        let context = container.newBackgroundContext()
        // [设计复盘] 为什么后台上下文需要单独设置 mergePolicy？
        // 答：后台上下文和 viewContext 的 mergePolicy 是独立的。
        // 如果后台上下文用默认的 error，保存冲突时会抛出异常，导致后台写入失败。
        context.mergePolicy = NSMergePolicy.mergeByPropertyObjectTrump
        return context
    }
}
```

**[设计复盘] 高频问题**："`NSManagedObjectContext` 为什么不是线程安全的？"

> 答：`NSManagedObject` 的生命周期与创建它的上下文绑定。Core Data 内部使用 `NSThread` 的 `threadDictionary` 来追踪上下文归属线程。跨线程访问会导致 `EXC_BAD_INSTRUCTION` 或数据损坏。正确做法是：
> - 主线程操作：`viewContext`（与主线程 RunLoop 关联）
> - 后台写入：`newBackgroundContext()` + `perform`
> - 线程间传递对象：只传 `NSManagedObjectID`，而不是对象本身

### 1.4 Value Transformer 与自定义类型存储

```swift
// [原理] StringArrayTransformer 继承 ValueTransformer，将 [String] 编码为 JSON Data。
// Core Data 的 Transformable 属性底层存储的是 BLOB（二进制大对象），
// 通过 Value Transformer 在读写时做编解码转换。
@objc(StringArrayTransformer)
final class StringArrayTransformer: ValueTransformer {
    override func transformedValue(_ value: Any?) -> Any? {
        guard let array = value as? [String] else { return nil }
        return try? JSONEncoder().encode(array)  // [String] → Data (JSON BLOB)
    }
    
    override func reverseTransformedValue(_ value: Any?) -> Any? {
        guard let data = value as? Data else { return nil }
        return try? JSONDecoder().decode([String].self, from: data)  // Data → [String]
    }
}

// 注册时机：DataController.init() 中，在 loadPersistentStores 之前执行
ValueTransformer.setValueTransformer(
    StringArrayTransformer(),
    forName: NSValueTransformerName("StringArrayTransformer")
)
```

**[设计复盘] 高频问题**："SwiftData 怎么处理 Transformable？"

> 答：SwiftData 的 `@Attribute(.externalStorage)` 和 `@Attribute(.allowsCloudEncryption)` 可以替代 Value Transformer，但底层实现仍然相同——`NSValueTransformer` 在 SwiftData 宏展开时自动合成。理解 Core Data 的 Value Transformer 机制，能解释 SwiftData 的"魔法"本质。

## 2. WidgetKit 原理深度解析

### 2.1 App Intent vs IntentConfiguration（iOS 17 范式迁移）

```swift
// [原理] iOS 17 之前用 IntentConfiguration，需要定义 Intent 继承 INIntent（IntentsKit 框架）。
// iOS 17 之后用 AppIntentConfiguration，Intent 继承 AppIntent（AppIntents 框架），
// 与 Spotlight、Shortcuts、Siri 统一了交互模型。

// 本项目使用 AppIntent（iOS 17+），因为 deployment target 是 iOS 16，但 Widget 扩展可以用 iOS 17 API。
// [设计复盘] 如果读者问"为什么不用 IntentConfiguration？"——答：AppIntent 是 Apple 推荐的未来方向，
// 与 Shortcuts 和 App Shortcuts 共享同一套类型系统，减少框架碎片化。

struct ExamCountdownIntent: AppIntent {
    static var title: LocalizedStringResource = "考试倒计时"
    
    // [原理] @Parameter 暴露可配置参数到 Widget 编辑界面
    @Parameter(title: "考试筛选", default: .all)
    var filter: ExamFilter
    
    enum ExamFilter: String, AppEnum {
        case all
        case nearest
        
        // [原理] TypeDisplayRepresentation 定义参数在 Widget 配置 UI 中的显示文本
        static var typeDisplayRepresentation: TypeDisplayRepresentation {
            TypeDisplayRepresentation(name: "考试筛选")
        }
    }
}
```

**[设计复盘] 高频问题**："`AppIntent` 和 `INIntent` 的本质区别是什么？"

> 答：
> 1. **框架归属**：`INIntent` 属于 `Intents` 框架（SiriKit 历史包袱），`AppIntent` 属于 `AppIntents` 框架（Swift 原生）
> 2. **类型系统**：`AppIntent` 支持 Swift 原生类型（`String`、`Enum`、`Codable`），`INIntent` 需要 `INSpeakableString` 等桥接类型
> 3. **平台统一**：`AppIntent` 同时支持 Widget、Shortcuts、Spotlight、Action Button，`INIntent` 仅限 Siri/Widget
> 4. **编译时安全**：`AppIntent` 的参数在编译期检查，`INIntent` 依赖运行时字典

### 2.2 Timeline 刷新机制：系统级电池优化策略

```swift
struct ExamCountdownProvider: AppIntentTimelineProvider {
    func timeline(for configuration: ExamCountdownIntent, in context: Context) async -> Timeline<ExamCountdownEntry> {
        let entry = buildEntry()
        
        // [原理] Timeline 是 WidgetKit 的核心抽象。系统不会实时运行 Widget 代码，
        // 而是预先生成一组 (entry, updatePolicy) 对，在特定时间点触发刷新。
        // 这本质上是"时间轴驱动的函数式渲染"：Widget = f(time, data)。
        
        // [原理] 考试倒计时 Widget 每天只需要在午夜刷新一次（天数变化）。
        // 使用 .after(nextMidnight) 告诉系统："到这个时间之前，不需要唤醒我"。
        var calendar = Calendar(identifier: .gregorian)
        calendar.timeZone = TimeZone(identifier: "Europe/London") ?? .current
        let nextMidnight = calendar.nextDate(
            after: Date(),
            matching: DateComponents(hour: 0, minute: 0),
            matchingPolicy: .nextTime
        ) ?? Date().addingTimeInterval(3600)
        
        return Timeline(entries: [entry], policy: .after(nextMidnight))
    }
}

// 对比其他 Widget 的刷新策略：
struct TodayTasksProvider: AppIntentTimelineProvider {
    func timeline(...) async -> Timeline<TodayTasksEntry> {
        // [原理] 任务状态变化更频繁，每小时刷新一次
        let nextUpdate = Calendar.current.date(byAdding: .hour, value: 1, to: Date())!
        return Timeline(entries: [entry], policy: .after(nextUpdate))
    }
}

struct DailyProgressProvider: AppIntentTimelineProvider {
    func timeline(...) async -> Timeline<DailyProgressEntry> {
        // [原理] 学习进度可能随时变化，每 30 分钟刷新
        let nextUpdate = Calendar.current.date(byAdding: .minute, value: 30, to: Date())!
        return Timeline(entries: [entry], policy: .after(nextUpdate))
    }
}
```

**[设计复盘] 高频问题**："Widget 刷新有频率限制吗？"

> 答：有，且非常严格。Apple 的**每日预算系统**（budget system）限制每个 Widget 的刷新次数：
> - 系统小部件（System Widget）：约 40-60 次/天
> - 用户频繁查看的 Widget 会获得更高预算
> - 使用 `WidgetCenter.shared.reloadTimelines(ofKind:)` 会消耗预算，不应在主 App 中频繁调用
> - 最佳实践：让 Timeline 的 `policy` 定义下一次刷新时间，而非主动 reload

### 2.3 App Group 数据共享原理：跨进程通信的底层实现

```swift
// [原理] App Group 是 Apple 提供的跨进程安全通信机制，底层基于 IPC（Inter-Process Communication）。
// 主 App 和 Widget Extension 运行在不同进程（不同沙盒），无法直接共享内存。
// App Group 通过创建共享容器目录（/private/var/mobile/Containers/Shared/AppGroup/...）
// 让两个进程读写同一个文件系统空间。

enum WidgetAppGroup {
    static let suiteName = "group.com.jeffhan.FinalPilot"
    
    static var defaults: UserDefaults? {
        // [原理] UserDefaults(suiteName:) 不是普通的 plist 存储，
        // 底层通过 CFPreferences 框架，以 Mach IPC 消息实现跨进程同步。
        // 当主 App 调用 set() 时，Widget 进程通过 notifyd（通知守护进程）收到变更通知。
        UserDefaults(suiteName: suiteName)
    }
}

// [原理] 数据流向：主 App → JSON 编码 → UserDefaults(suiteName) → Widget 读取 → JSON 解码
extension FinalPilotStore {
    func syncToWidget() {
        let provider = WidgetDataProvider.shared
        
        // 1. 主 App 将数据编码为 JSON
        let examInfos = courses.compactMap { course -> WidgetExamInfo? in
            guard let examDate = course.examDate else { return nil }
            return WidgetExamInfo(id: course.id, name: course.name, examDate: examDate, ...)
        }
        provider.saveExams(examInfos)  // 内部：JSONEncoder().encode(examInfos) → UserDefaults.set(data, forKey:)
        
        // 2. 触发 Widget 刷新（消耗预算，只在关键数据变化时调用）
        // WidgetCenter.shared.reloadTimelines(ofKind: "ExamCountdownWidget")
    }
}
```

**[设计复盘] 高频问题**："App Group 和 App Extension 的生命周期关系？"

> 答：
> 1. **进程隔离**：主 App 和 Widget Extension 是**两个独立的进程**，由 `pkd`（Plug-in Kit Daemon）管理。Widget 进程在以下场景启动：
>    - 用户添加到主屏幕/锁屏
>    - Timeline 刷新时间到达
>    - 主 App 调用 `reloadTimelines`
> 2. **内存限制**：Widget 进程内存上限约 16-24MB（iOS 17），远低于主 App（数百 MB）。因此 Widget 不能加载 Core Data 完整栈。
> 3. **启动速度**：Widget 必须在**1-2 秒**内完成渲染，所以使用预序列化的 JSON 数据而非实时查询 Core Data。

### 2.4 Widget 与主 App 生命周期关系

```swift
// [原理] FinalPilotApp.swift 中，Widget 数据同步发生在两个时机：
@main
struct FinalPilotApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .onAppear {
                    // 时机1：首次启动，数据迁移完成后立即同步
                    store.syncToWidget()
                }
                .onReceive(store.objectWillChange) { _ in
                    // 时机2：任何数据变化时自动同步（通过 ObservableObject 的 willChange 通知）
                    // [原理] objectWillChange 是 Combine 的 Publisher，在 @Published 属性变化前发出事件。
                    // 这里利用它来捕获所有数据变化，避免在每个 setter 中手动调用 syncToWidget()。
                    store.syncToWidget()
                }
        }
    }
}
```

**[设计复盘] 高频问题**："Widget 能直接访问主 App 的 Core Data 吗？"

> 答：**不能**。Widget 进程无法直接访问主 App 的 `NSPersistentContainer`。解决方案有三：
> 1. **App Group + UserDefaults**（本项目采用）：简单数据，JSON 序列化
> 2. **App Group + Core Data with NSPersistentStoreCoordinator**：复杂数据，但 Widget 内存限制可能导致 OOM
> 3. **Background URLSession + CloudKit**：实时性要求高的场景，Widget 通过 CloudKit 查询

## 3. 本地通知系统原理

### 3.1 UNUserNotificationCenter 权限生命周期

```swift
final class NotificationManager: ObservableObject {
    @Published var authorizationStatus: UNAuthorizationStatus = .notDetermined
    
    func requestAuthorization() {
        // [原理] requestAuthorization 第一次调用时弹出系统弹窗，后续调用直接返回已授权状态。
        // 弹窗的显示由 SpringBoard（系统桌面进程）控制，App 无法自定义弹窗内容或样式。
        center.requestAuthorization(options: [.alert, .badge, .sound]) { granted, error in
            // [原理] 回调在后台线程执行，必须切回主线程更新 UI 绑定的 @Published 属性。
            DispatchQueue.main.async {
                self.authorizationStatus = granted ? .authorized : .denied
            }
        }
    }
    
    func checkAuthorizationStatus() {
        // [原理] getNotificationSettings 是异步查询，读取的是 SpringBoard 维护的权限数据库，
        // 不是内存中的状态。用户可能在设置中手动关闭权限，所以需要每次启动时检查。
        center.getNotificationSettings { [weak self] settings in
            DispatchQueue.main.async {
                self?.authorizationStatus = settings.authorizationStatus
                self?.isNotificationsEnabled = settings.authorizationStatus == .authorized
            }
        }
    }
}
```

**[设计复盘] 高频问题**："用户首次拒绝通知权限后，如何再次引导开启？"

> 答：
> 1. **不能直接再次弹出系统弹窗**：iOS 只给一次弹窗机会，再次调用 `requestAuthorization` 不会显示弹窗。
> 2. **引导到系统设置**：通过 `UIApplication.openSettingsURLString` 打开设置页面，让用户手动开启。
> 3. **软引导策略**：在 App 内显示自定义引导页面（如 FinalPilot 的"开启通知获得复习提醒"），说明价值后再跳转设置。

### 3.2 通知分类（Category）与 Actionable 通知

```swift
private func registerNotificationCategories() {

    let markDoneAction = UNNotificationAction(

    let remindLaterAction = UNNotificationAction(
```

**[设计复盘] 高频问题**："用户点击通知 Action 后，App 没启动，怎么执行逻辑？"

> 答：通过 `UNUserNotificationCenterDelegate` 的 `didReceive response` 方法处理：
> ```swift
> func userNotificationCenter(
>     _ center: UNUserNotificationCenter,
>     didReceive response: UNNotificationResponse,
>     withCompletionHandler completionHandler: @escaping () -> Void
> ) {
>     let actionIdentifier = response.actionIdentifier
>     // 即使是后台操作，系统也会短暂唤醒 App 进程执行这个回调
>     // 但执行时间有限制（约 30 秒），超时会被系统终止
>     switch actionIdentifier {
>     case "MARK_DONE":
>         // 在后台标记任务完成，可能需要 Core Data 后台上下文写入
>     case "REMIND_LATER":
>         // 调度一个新的延迟通知
>     }
>     completionHandler()
> }
> ```

### 3.3 通知调度类型与后台限制

```swift
// 类型1：每日重复提醒（UNCalendarNotificationTrigger）
func scheduleDailyReminder(hour: Int, minute: Int = 0) {
    var components = DateComponents()
    components.hour = hour
    components.minute = minute
    // [原理] repeats: true 表示每天同一时间触发。系统将其注册到 SpringBoard 的闹钟服务中，
    // 即使 App 被杀死，iOS 也会在指定时间唤醒系统服务发送通知。
    let trigger = UNCalendarNotificationTrigger(dateMatching: components, repeats: true)
}

// 类型2：一次性倒计时通知（UNCalendarNotificationTrigger）
func scheduleOneTimeReminder(date: Date) {
    let triggerComponents = Calendar.current.dateComponents([.year, .month, .day, .hour, .minute], from: date)
    // [原理] repeats: false，通知触发后自动从系统队列中移除
    let trigger = UNCalendarNotificationTrigger(dateMatching: triggerComponents, repeats: false)
}

// 类型3：错题回顾延迟通知（UNTimeIntervalNotificationTrigger）
func scheduleMistakeReviewReminder(afterMinutes: Int) {
    // [原理] timeInterval 从调用 add() 的时刻开始倒计时。如果设备重启，倒计时会重新计算。
    let trigger = UNTimeIntervalNotificationTrigger(timeInterval: TimeInterval(afterMinutes * 60), repeats: false)
}
```

**[设计复盘] 高频问题**："iOS 通知的优先级是怎样的？本地通知和远程通知哪个优先级高？"

> 答：iOS 15 引入的通知优先级系统（ interruption levels）：
> 1. **Passive**（被动）：不亮屏、不播放声音、不显示横幅，仅在通知中心可见
> 2. **Active**（默认）：亮屏、播放声音、显示横幅（本项目未设置，默认此级别）
> 3. **Time Sensitive**（时间敏感）：可突破专注模式（Focus Mode）限制，但需申请权限
> 4. **Critical**（紧急）：无视静音开关和专注模式，仅限医疗、安全等场景，需特殊权限
> 
> 本地通知和远程通知的**系统调度优先级相同**，但远程通知有额外的 APNs 推送延迟（通常 1-3 秒）。

## 4. 艾宾浩斯遗忘曲线算法

### 4.1 数学公式推导

艾宾浩斯遗忘曲线（Ebbinghaus Forgetting Curve）的数学模型：

```
R(t) = e^(-t / S)
```

其中：
- `R(t)`：记忆保留率（0 到 1 之间）
- `t`：自学习以来经过的时间
- `S`：记忆强度（与复习次数正相关，首次学习 S ≈ 1 天）

**[原理] 在间隔重复系统中，我们不需要精确计算 R(t)，而是使用固定的最佳复习间隔**。这些间隔是基于艾宾浩斯实验数据的经验值：

| 复习阶段 | 间隔时间 | 记忆保留率（估算） | 算法用途 |
|---------|---------|------------------|---------|
| 第1次 | 20 分钟 | ~58% | 立即巩固 |
| 第2次 | 1 小时 | ~44% | 短期记忆固化 |
| 第3次 | 9 小时 | ~36% | 日间复习 |
| 第4次 | 1 天 | ~34% | 日结复习 |
| 第5次 | 2 天 | ~28% | 周间复习 |
| 第6次 | 6 天 | ~25% | 周复习 |
| 第7次 | 31 天 | ~21% | 月复习 |

```swift
// [原理] 将间隔转换为 TimeInterval（秒）
let intervals: [TimeInterval] = [
    20 * 60,      // 20 分钟 = 1200 秒
    60 * 60,      // 1 小时 = 3600 秒
    9 * 60 * 60,  // 9 小时 = 32400 秒
    86400,        // 1 天 = 86400 秒
    172800,       // 2 天 = 172800 秒
    518400,       // 6 天 = 518400 秒
    2678400       // 31 天 ≈ 2678400 秒（按 30.44 天平均月）
]
```

### 4.2 节点计算与逾期判断逻辑

```swift
static func ebbinghausReviewSuggestions(attempts: [QuizAttempt]) -> [EbbinghausSuggestion] {
    let intervals: [TimeInterval] = [20 * 60, 60 * 60, 9 * 60 * 60, 86400, 172800, 518400, 2678400]
    let intervalLabels = ["20分钟后", "1小时后", "9小时后", "1天后", "2天后", "6天后", "31天后"]
    
    // [原理] tolerance = 1.5 表示在"目标间隔 × 1.5"范围内都算作该阶段的复习窗口。
    // 例如第1阶段（20分钟）：窗口为 [20分钟, 30分钟)。如果超过 30 分钟，进入第2阶段窗口。
    // 这种"模糊匹配"避免用户因错过精确时间点而遗漏复习。
    let tolerance: Double = 1.5
    
    let wrongAttempts = attempts.filter { !$0.isCorrect }
    var suggestions: [EbbinghausSuggestion] = []
    
    for attempt in wrongAttempts {
        // [原理] elapsed = 当前时间 - 错题创建时间。这是判断"处于哪个遗忘阶段"的核心输入。
        let elapsed = Date().timeIntervalSince(attempt.createdAt)
        
        for (index, interval) in intervals.enumerated() {
            // [原理] 匹配逻辑：elapsed >= interval 且 elapsed < interval × tolerance
            // 即：刚好进入该阶段，且尚未超过该阶段窗口。
            if elapsed >= interval && elapsed < interval * tolerance {
                // [原理] isOverdue = elapsed > interval × 2，表示已经超过该阶段最佳复习窗口的两倍时间。
                // 例如第1阶段：超过 40 分钟（20×2）即标记为逾期，需要优先复习。
                let isOverdue = elapsed > interval * 2
                
                suggestions.append(EbbinghausSuggestion(
                    questionID: attempt.questionID,
                    knowledgePointID: attempt.knowledgePointID,
                    stage: index,                       // [设计复盘] stage 从 0 开始，对应第1次复习
                    stageLabel: intervalLabels[index],
                    elapsedHours: elapsed / 3600,
                    isOverdue: isOverdue
                ))
                break  // [原理] 每个错题只匹配一个阶段，匹配到即跳出
            }
        }
    }
    
    // [原理] 按 stage 排序：优先推荐早期阶段的复习（逾期风险更高）
    return suggestions.sorted { $0.stage < $1.stage }
}
```

**[设计复盘] 高频问题**："为什么 tolerance 设为 1.5，而不是精确的 1.0？"

> 答：
> 1. **用户行为不确定性**：用户不可能在精确的 20 分钟或 1 小时整点打开 App。1.5 倍容忍度给予 50% 的弹性时间。
> 2. **避免阶段跳跃**：如果容忍度是 1.0，用户在第 21 分钟打开时，第1阶段刚结束，系统会立即跳到第2阶段（1小时），导致第1阶段复习被跳过。
> 3. **心理学依据**：艾宾浩斯原始实验中的时间记录也是近似的，间隔重复算法（如 SuperMemo）通常使用 1.2-2.0 的弹性系数。

### 4.3 与 Anki SM-2 算法的对比

| 维度 | FinalPilot 简化版 | Anki SM-2 |
|------|------------------|-----------|
| **间隔计算** | 固定经验值（20min/1h/9h/...） | 基于用户评分动态调整 |
| **记忆难度** | 二值（错/对） | 5 级评分（Again/Hard/Good/Easy） |
| **EF 因子** | 无 |  easiness factor (初始 2.5，根据评分调整) |
| **实现复杂度** | 20 行代码 | 50+ 行，含大量参数调优 |
| **适用场景** | 考试冲刺（短期密集） | 长期语言学习（数年） |

```swift
// [原理] Anki SM-2 的核心公式（供设计复盘对比引用）：

// 其中 EF（easiness factor）根据用户每次回答的自我评估调整：

//     3. 固定间隔在短期冲刺中已足够有效，且代码更简洁、bug 更少。
```

## 5. SwiftUI 暗色模式实现原理

### 5.1 UIColor dynamicProvider 底层机制

```swift
// [原理] UIColor { traitCollection in ... } 创建的是"动态颜色"（Dynamic Color），
// 它不是两个静态颜色的简单封装，而是一个闭包，在每次渲染时根据当前 traitCollection 动态解析。
// 这利用了 iOS 的 UITraitCollection 系统，它是响应式 UI 的底层基础设施。

extension Color {
    init(light: UIColor, dark: UIColor) {
        self.init(uiColor: UIColor { traitCollection in
            switch traitCollection.userInterfaceStyle {
            case .dark:
                return dark
            default:
                return light
            }
        })
    }
}

// [原理] traitCollection 是 UIView/UIViewController 的一个属性，描述当前环境的视觉特征：
// - userInterfaceStyle（.light / .dark）
// - displayGamut（sRGB / P3）
// - horizontalSizeClass（compact / regular）
// - displayScale（@1x / @2x / @3x）
// 当用户在控制中心切换暗色模式时，系统会遍历所有可见视图的 traitCollection，触发重新渲染。
```

**[设计复盘] 高频问题**："`UIColor dynamicProvider` 和 Asset Catalog 的 Color Set 有什么区别？"

> 答：
> 1. **实现方式**：Color Set 在编译期生成 `.car` 文件（Compiled Asset Archive），运行时通过 `UIColor(named:)` 加载；`dynamicProvider` 是运行时闭包执行。
> 2. **性能**：Color Set 略快，因为 `.car` 文件是内存映射的；`dynamicProvider` 每次调用有闭包开销，但在现代设备上可忽略。
> 3. **灵活性**：`dynamicProvider` 可以在闭包中做任意逻辑（如根据时间、位置选择颜色），Color Set 只能是静态的 light/dark 值。
> 4. **适用场景**：Color Set 适合设计规范中固定的颜色；`dynamicProvider` 适合代码中需要动态计算的颜色（如根据主题色生成渐变）。

### 5.2 traitCollection 变化监听与响应式更新

```swift
// [原理] SwiftUI 中监听暗色模式变化有三种方式：

            .background(AppTheme.adaptiveBackground(forScheme: colorScheme))

// 方式2：UIColor dynamicProvider —— UIKit 桥接（见上文 init(light:dark:)）
// 适用场景：需要将颜色传递给 UIKit 组件（如 UIViewRepresentable）或 Core Graphics 绘制时。

// 方式3：UIUserInterfaceStyle 强制覆盖
// .preferredColorScheme(.dark) 强制视图以暗色模式渲染，不受系统设置影响。
```

**[设计复盘] 高频问题**："用户在控制中心切换暗色模式，App 如何收到通知？"

> 答：
> 1. **SwiftUI 自动处理**：`@Environment(\.colorScheme)` 的视图会自动重新渲染，无需手动监听。
> 2. **UIKit 手动监听**：通过 `override func traitCollectionDidChange(_ previousTraitCollection: UITraitCollection?)` 在 UIViewController 中捕获变化。

### 6.2 Canvas 粒子系统：确定性随机与物理模拟

```swift
struct ConfettiView: View {
    var body: some View {
        TimelineView(.animation(minimumInterval: 1.0 / 60.0)) { timeline in
            let elapsed = timeline.date.timeIntervalSince(startTime)
            Canvas { context, size in
                // [原理] 使用伪随机数生成器（LCG 线性同余法），确保同一帧的粒子位置可重现。
                // 与 arc4random() 不同，LCG 是确定性的：相同种子产生相同序列，避免粒子闪烁。
                var seed = startTime.hashValue
                func nextRandom() -> Double {
                    seed = seed &* 6364136223846793005 &+ 1  // [原理] LCG 经典参数：a = 6364136223846793005, c = 1
                    return Double(UInt64(bitPattern: Int64(seed))) / Double(UInt64.max)
                }
                
                for _ in 0..<particleCount {
                    let t = CGFloat(elapsed)
                    
                    // [原理] 物理模拟：抛物线运动 y = y0 + vy * t + 0.5 * g * t²
                    // 这里 g = 400（像素/秒²），是简化重力加速度。
                    let x = startX + vx * t * 30
                    let y = startY + vy * t * 30 + 0.5 * 400 * t * t
                    
                    // [原理] 衰减函数：opacity = 1 - t / (decay * 2.5)，线性淡出
                    let opacity = max(0, 1.0 - Double(t) / Double(decay * 2.5))
                    
                    // [原理] 边界裁剪：粒子超出视图范围时跳过绘制，减少 GPU 开销
                    guard opacity > 0, y < size.height + 50 else { continue }
                    
                    context.fill(Path(rect.applying(transform)), with: .color(color.opacity(opacity)))
                }
            }
        }
    }
}

// [设计复盘] 为什么用 Canvas 而不是 ZStack + ForEach？
// 答：Canvas 使用 Core Graphics 直接绘制，绕过了 SwiftUI 的视图树布局系统。
// 50 个粒子用 ZStack 会创建 50 个视图节点，布局计算开销 O(n)；
// Canvas 直接绘制到 CALayer，开销 O(1)（对 GPU 而言）。
```

### 6.3 Animatable Protocol：自定义属性插值

```swift
// [原理] Animatable 是 SwiftUI 的动画核心协议。它定义了 animatableData 属性，
// SwiftUI 的动画引擎在动画过程中不断修改这个值，视图根据新值重新渲染。
// 本质是"属性的数值插值"：引擎知道 Double 如何插值（线性或缓动），但不知道你的业务含义。

struct AnimatedNumber: View, Animatable {
    var value: Double
    
    // [原理] animatableData 是动画引擎的"操作手柄"。
    // 当 withAnimation 改变 value 时，引擎实际修改的是 animatableData，
    // 然后触发 body 的重新计算，显示新的 Text。
    var animatableData: Double {
        get { value }
        set { value = newValue }
    }
    
    var body: some View {
        Text(formatter(value))
            .monospacedDigit()  // [原理] 等宽数字防止动画时文字跳动
    }
}

// 使用方式：
AnimatedNumber(value: displayedValue)
    .onAppear {
        withAnimation(.easeOut(duration: 1.0)) {
            displayedValue = targetValue  // 动画引擎从 0 插值到 targetValue，持续 1 秒
        }
    }

// [设计复盘] 如果 value 是 String 类型，能实现 Animatable 吗？
// 答：不能直接实现。Animatable 要求 animatableData 是 VectorArithmetic 类型（如 Double、CGFloat、CGPoint）。
// 字符串不是数值，无法插值。解决方案：将字符串动画拆分为多个阶段（如逐字显示），
// 或转换为数值（如字符索引），动画数值后再映射回字符串。
```

### 6.4 GeometryEffect：自定义几何变换动画

```swift
// [原理] GeometryEffect 是 SwiftUI 最底层的动画协议，直接操作视图的投影变换矩阵（ProjectionTransform）。
// 它继承自 Animatable，因此可以参与动画插值。

struct ShakeEffect: GeometryEffect {
    var amount: CGFloat = 8
    var shakes: CGFloat = 3
    var animatableData: CGFloat  // 动画引擎插值的"进度值"（0→1）
    
    var multiplier: CGFloat { 2 * .pi * shakes }  // 完整振动周期数
    
    func effectValue(size: CGSize) -> ProjectionTransform {
        // [原理] 正弦波振动：x = sin(2π * shakes * progress) * amount
        // 当 progress 从 0→1，sin 函数完成 shakes 个完整周期，产生左右振动效果。
        let x = sin(animatableData * multiplier) * amount
        return ProjectionTransform(CGAffineTransform(translationX: x, y: 0))
    }
}

// 使用方式：
view.shake(trigger: errorOccurred)  // 当 errorOccurred 变为 true，动画从 0 到 1

// [设计复盘] GeometryEffect 与 ViewModifier 的动画有什么区别？
// 答：ViewModifier 只能修改视图属性（如 offset、scale），这些属性的动画由 SwiftUI 内置支持。
// GeometryEffect 直接操作 Core Animation 的变换矩阵，可以实现任意几何变换（如扭曲、波浪、3D 翻转），
// 这是 ViewModifier 无法做到的。
```

```swift
    // 当数据点稀疏时，用户可以通过点精确定位数值。
    PointMark(
        x: .value("日期", record.date, unit: .day),
        y: .value("分钟", record.studyMinutes)
    )
    .foregroundStyle(AppTheme.primary)
    .symbolSize(60)
}

// [设计复盘] Charts 框架和 UIKit 的 Core Plot / Charts（danielgindi）有什么区别？
// 答：
// 1. Charts 是 Apple 原生框架，与 SwiftUI 深度集成，支持自动暗色模式、动画、可访问性。
// 2. 第三方 Charts 库基于 Core Graphics 手动绘制，需要处理布局、手势、动画，代码量大。
// 3. Charts 的 API 是声明式的，与 SwiftUI 的 View 语法一致；第三方库是命令式的。
// 4. 性能上，Apple Charts 在数据量 < 10,000 时足够高效，大数据量仍需第三方库或自定义绘制。
```

### 7.2 Catmull-Rom 插值原理

```swift
.interpolationMethod(.catmullRom)

// [原理] Catmull-Rom 插值是一种样条插值算法，通过所有数据点生成平滑曲线。
// 与线性插值（linear）和单调插值（monotone）不同，Catmull-Rom 会穿过每个数据点，
// 并通过控制点（前后相邻点）计算切线，使曲线在连接处 C1 连续（一阶导数连续）。

// 数学公式（简化版）：
// 给定四个连续点 P0, P1, P2, P3，在 P1 和 P2 之间生成曲线段。
// 参数方程：
// CR(t) = 0.5 * [(2P1) + (-P0 + P2) * t + (2P0 - 5P1 + 4P2 - P3) * t² + (-P0 + 3P1 - 3P2 + P3) * t³]
// 其中 t ∈ [0, 1]

// [设计复盘] 为什么用 Catmull-Rom 而不是 .linear？
// 答：
// 1. 线性插值在数据点处形成"折线"，视觉上不够平滑，暗示数据有突变。
// 2. 学习时长趋势是连续过程，Catmull-Rom 的平滑曲线更符合用户的心理模型。
// 3. 但 Catmull-Rom 可能产生"过冲"（overshoot），即曲线超出数据点范围。如果数据必须有物理意义（如非负），
//    应使用 .monotone（单调插值），它保证曲线不会超出数据范围。
```

### 7.3 自定义饼图与渐变色填充

```swift
// [原理] 本项目中的知识点分布图不是用 Charts 的 SectorMark（iOS 17+），
// 而是自定义 Shape + GeometryReader，因为 deployment target 是 iOS 16，SectorMark 不可用。

private struct PieSliceShape: Shape {
    let startAngle: Double
    let endAngle: Double
    
    func path(in rect: CGRect) -> Path {
        var path = Path()
        let center = CGPoint(x: rect.midX, y: rect.midY)
        let radius = min(rect.width, rect.height) / 2
        
        path.move(to: center)
        path.addArc(
            center: center,
            radius: radius,
            startAngle: .degrees(startAngle),
            endAngle: .degrees(endAngle),
            clockwise: false  // [原理] false = 逆时针方向，与数学惯例一致
        )
        path.closeSubpath()  // 闭合到圆心，形成扇形
        return path
    }
}

// [原理] 环形图（Donut Chart）通过叠加白色中心圆实现：
ZStack {
    ForEach(0..<knowledgeDistribution.count, id: \.self) { index in
        let startAngle = knowledgeDistribution.prefix(index).reduce(0) { $0 + $1.percentage } * 360 - 90
        let endAngle = startAngle + knowledgeDistribution[index].percentage * 360
        PieSliceShape(startAngle: startAngle, endAngle: endAngle)
            .fill(statusColor(knowledgeDistribution[index].status))
    }
    Circle()
        .fill(AppTheme.card)  // 中心圆覆盖，形成环形效果
        .frame(width: size * 0.45, height: size * 0.45)
}
```

## 8. CI/CD 工程化原理

### 8.1 GitHub Actions Runner 机制

```yaml
# [原理] GitHub Actions 使用事件驱动的 workflow 模型。
# 当满足触发条件（on.push / on.pull_request）时，GitHub 的调度器分配一个 runner（虚拟机）执行 job。
# runner 是临时容器：每次执行后销毁，不保留状态（除非使用 cache）。

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-lint:
    runs-on: macos-15  # [原理] macOS runner 是 Apple Silicon 虚拟机（M1/M2），由 GitHub 托管
    timeout-minutes: 30  # [原理] 超时防止资源浪费：如果构建死锁，30 分钟后自动终止
```

**[设计复盘] 高频问题**："GitHub Actions 的 runner 有哪些类型？"

> 答：
> 1. **GitHub-hosted runner**：由 GitHub 提供，免费额度：
>    - 公共仓库：无限制（但并发 job 限制为 20 个）
>    - 私有仓库：2,000 分钟/月（Linux），200 分钟/月（macOS）
> 2. **Self-hosted runner**：在自己的机器上运行，适合需要特定硬件（如真机测试）的场景
> 3. **macOS runner 的特殊性**：Apple 的 EULA 限制，macOS 虚拟机只能在 Apple 硬件上运行，因此 GitHub 的 macOS runner 成本是 Linux 的 10 倍

### 8.2 xcodebuild 构建流程

```yaml
      - name: 构建验证（xcodebuild）
        run: |
          # [原理] xcodebuild 是 Xcode 的命令行构建工具，底层调用 LLVM/Clang 编译器和 ld64 链接器。
          # 构建流程：编译 .swift → 生成 .o（目标文件）→ 链接 .o → 生成 .app → 签名（可选）→ 生成 .ipa
          
          # 步骤1：查找可用模拟器
          DEVICE_ID=$(xcrun simctl list devices available | grep -i "iPhone" | grep -v "Plus" | head -1 | grep -oE '[A-F0-9-]{36}')
          
          # 步骤2：构建
          xcodebuild             -project FinalPilotApp.xcodeproj             -scheme FinalPilotApp             -configuration Debug             -sdk iphonesimulator \  # [原理] 使用模拟器 SDK，无需代码签名和证书
            -destination "$DESTINATION"             -derivedDataPath ${{ runner.temp }}/DerivedData \  # [原理] DerivedData 缓存编译产物，加速增量构建
            CODE_SIGNING_ALLOWED=NO \  # [原理] 禁用代码签名，因为 CI 环境没有开发者证书私钥
            build || true  # [原理] || true 允许构建失败时继续执行（如仅做编译检查，不阻断流程）
```

**[设计复盘] 高频问题**："`xcodebuild` 和 `swift build` 有什么区别？"

> 答：
> 1. **xcodebuild**：调用 Xcode 的构建系统，处理 `.xcodeproj`、`.xcworkspace`、`Info.plist`、Asset Catalog、故事板等，适合 iOS/macOS App。
> 2. **swift build**：Swift Package Manager 的构建命令，处理 `Package.swift`，适合纯 Swift 库/CLI 工具。
> 3. **混合使用**：如果项目同时使用 Xcode 和 SPM，用 `xcodebuild` 构建时，SPM 依赖会自动解析（通过 Xcode 的 Package Resolution 阶段）。

### 8.3 并发控制与资源优化

```yaml
# [原理] concurrency 配置防止同一分支的多个 workflow 同时运行，浪费资源。
# 典型场景：开发者快速 push 多次，旧 workflow 还在运行，新 workflow 又启动。
# cancel-in-progress: true 会自动取消旧的 workflow，只保留最新的。
concurrency:
  group: ci-${{ github.ref }}  # [原理] group 按分支分组，不同分支互不干扰
  cancel-in-progress: true
```

**[设计复盘] 高频问题**："CI 中构建失败怎么排查？"

> 答：
> 1. **查看日志**：GitHub Actions 的日志按步骤折叠，先定位失败步骤
> 2. **本地复现**：在本地执行相同的 `xcodebuild` 命令，排除环境差异
> 3. **DerivedData 清理**：有时缓存损坏导致编译错误，CI 中应定期清理 `-derivedDataPath`
> 4. **SwiftLint 版本**：CI 安装的 SwiftLint 版本可能与本地不同，导致规则差异。应在 `.swiftlint.yml` 中锁定版本或禁用新规则

## 附录：设计复盘速查卡

### Core Data
- **为什么用 count 而不是 fetch？** `count(for:)` 执行 SQL COUNT(*)，不加载行数据，O(1) 内存
- **mergePolicy 作用？** 解决对象图冲突，mergeByPropertyObjectTrump 以内存值优先
- **后台上下文怎么创建？** `newBackgroundContext()` + `perform` 块，保存后自动合并到 viewContext

### WidgetKit
- **Timeline 刷新频率？** 由系统预算控制，约 40-60 次/天，优先用 `policy` 而非 `reloadTimelines`
- **App Intent 和 INIntent 区别？** AppIntent 是 Swift 原生类型，统一 Widget/Shortcuts/Siri 模型
- **Widget 能访问 Core Data 吗？** 不能，需通过 App Group + UserDefaults 共享序列化数据

### 通知系统
- **权限请求后拒绝怎么办？** 只能引导到系统设置，无法再次弹窗
- **前台通知怎么显示？** 实现 `UNUserNotificationCenterDelegate.willPresent`
- **Time Sensitive 是什么？** 可突破专注模式，但需申请 entitlement

### 艾宾浩斯算法
- **公式？** `R(t) = e^(-t/S)`，固定间隔用于简化实现
- **tolerance 1.5 的作用？** 给予用户弹性时间，避免精确时间匹配导致的阶段跳跃
- **与 SM-2 对比？** 固定间隔适合短期冲刺，SM-2 适合长期学习（动态调整 EF）

### 暗色模式
- **UIColor dynamicProvider 原理？** 闭包在每次渲染时根据 traitCollection 解析颜色
- **ColorScheme 和 UIUserInterfaceStyle？** 同一概念的 SwiftUI 和 UIKit 表达
- **强制暗色模式？** `.preferredColorScheme(.dark)`

### SwiftUI 动画
- **TimelineView 和 withAnimation 区别？** TimelineView 是显式时间采样，withAnimation 是属性插值
- **Canvas 为什么性能高？** 直接 Core Graphics 绘制，绕过 SwiftUI 视图树布局
- **Animatable 原理？** 动画引擎修改 `animatableData`，触发视图重新渲染

### Charts
- **LineMark 数据绑定？** `x: .value("标签", data)` 自动映射到坐标轴
- **Catmull-Rom 插值？** 穿过所有数据点的平滑样条，视觉上比线性插值更连续
- **为什么自定义饼图？** iOS 16 不支持 SectorMark，用 Shape + Path 兼容旧版本

### CI/CD
- **xcodebuild 流程？** 编译 → 链接 → 签名（可选）→ 生成 .app
- **并发控制？** `concurrency` + `cancel-in-progress` 防止资源浪费
- **缓存策略？** DerivedData 缓存编译产物，key 需包含源文件 hash 保证正确性
